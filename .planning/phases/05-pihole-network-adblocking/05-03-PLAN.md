<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 4: Verify fallback DNS behavior</name>
  <what-built>
Pi-hole network-wide ad blocking with DHCP-provided DNS and fallback to 1.1.1.1 if Pi-hole is unavailable.
  </what-built>
  <how-to-verify>
**Test fallback DNS works when Pi-hole is stopped:**

1. **From a device that received DHCP from Pi-hole**, verify current state:
   - Open a browser, visit https://ads-blocker.com/testing/ or similar ad test page
   - Note that ads are blocked (Pi-hole working)

2. **Stop Pi-hole container:**
   ```bash
   cd apps/pihole
   docker compose stop pihole
   ```

3. **Wait 2-3 seconds, then test internet:**
   - Refresh web page - internet should still work
   - DNS is now going through 1.1.1.1 fallback
   - Ads may now appear (expected - fallback DNS doesn't block)

4. **Start Pi-hole again:**
   ```bash
   docker compose start pihole
   ```

5. **Verify Pi-hole is operational:**
   - Wait 10 seconds
   - Refresh page - ads should be blocked again
   - Check Pi-hole query log shows traffic

**Expected behavior:**
- When Pi-hole stops: ~1 second delay, then DNS works via 1.1.1.1
- When Pi-hole starts: DNS resumes through Pi-hole, ad blocking resumes
- No manual intervention required on client devices
  </how-to-verify>
  <resume-signal>Type "fallback verified" if internet works when Pi-hole is stopped, or describe any issues</resume-signal>
</task>
